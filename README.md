# Поисковик строительных материалов на базе RAG (ÖKOBAUDAT)

## Зачем вообще нужен такой поисковик

При подборе строительных материалов обычно нужно быстро ответить на практический вопрос:
**«Какие материалы подходят под мой запрос и как они выглядят по экологическим показателям?»**

В исходных данных ÖKOBAUDAT информации много, но в «сыром» виде ей трудно пользоваться:

- по одному материалу есть несколько записей по стадиям жизненного цикла (`A1`, `A2`, `A3`, `B`, `C`, `D`);
- числовые поля записаны в смешанных форматах (в том числе с запятой как десятичным разделителем);
- текст неоднородный, местами много «технического шума»;
- напрямую искать релевантные материалы неудобно, особенно если запрос сформулирован человеческим языком.

Поэтому в этом проекте реализован **retrieval-first подход**:

- система **не выдумывает факты**, а ищет по подготовленной базе;
- ответ можно объяснить через прозрачные баллы и метаданные;
- результаты можно дополнительно упорядочить с учетом бизнес-логики и эко-приоритета (по `GWP`).

Итог: получается практичный прототип «умного поиска» по материалам для сравнения и предварительного отбора.

---

## Цель проекта

Собрать воспроизводимый пайплайн от «сырого» CSV до интерактивного поиска:

1. подготовить данные в удобном для retrieval формате;
2. реализовать и сравнить несколько стратегий поиска (лексический, семантический, гибридный);
3. оценить качество на фиксированном наборе запросов;
4. показать работу в виде чат-интерфейса.

## Что и зачем делается: пошагово

### Шаг 1. Подготовка сырых данных

Ноутбук: `RAG_data_preparation_from_obd.ipynb`

Что делается:

- читается исходный CSV с корректными параметрами;
- удаляются шумные/технические части (например, `template dataset`, пустые служебные колонки);
- числовые поля нормализуются к единому формату (`float`).

Зачем:

- без очистки модель поиска будет «цепляться» за мусор;
- без нормализации чисел нельзя корректно сравнивать экологические показатели.

### Шаг 2. Приведение к сущности «один материал = одна запись»

Что делается:

- группировка по `UUID` (материалу);
- объединение метаданных в одну строку;
- выбор ключевых экологических индикаторов для сравнения (в первую очередь производственная стадия `A1-A3`);
- отбор записей, где есть валидные значения `GWP` (или fallback-поле).

Зачем:

- retrieval работает лучше, когда каждый документ соответствует одной смысловой сущности;
- сравнение материалов становится интерпретируемым и стабильным.

### Шаг 3. Формирование двух рабочих таблиц

Что делается:

- создается `df_text.csv` для поиска по тексту и эмбеддингам;
- создается `df_numeric.csv` для числовых фильтров и переранжирования.

Зачем два слоя:

- текстовый слой отвечает за «найти похожее по смыслу»;
- числовой слой отвечает за «сравнить и отсортировать по критериям».

### Шаг 4. Создание набора для оценки качества

Что делается:

- формируются тестовые запросы (`eval_queries.csv`);
- формируются пары релевантности (`eval_qrels.csv`) по правилам для основных классов материалов.

Зачем:

- без фиксированного evaluation-сета нельзя объективно сравнивать методы;
- метрики становятся воспроизводимыми между запусками.

### Шаг 5. Семантический индекс (Dense Retrieval)

Ноутбук: `RAG_material_search_oekobaudat.ipynb`

Что делается:

- тексты кодируются моделью `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`;
- строятся индексы FAISS:
  - `IndexFlatIP`,
  - `IndexIVFFlat`,
  - `IndexHNSWFlat`.

Зачем:

- dense-поиск хорошо ловит смысловые совпадения, даже если слова в запросе и документе различаются;
- разные индексы FAISS позволяют балансировать между скоростью и точностью.

### Шаг 6. Лексический базовый поиск (BM25)

Что делается:

- отдельно строится BM25 как keyword-baseline.

Зачем:

- BM25 часто сильнее на точных терминах/формулировках;
- baseline нужен для честного сравнения с семантическим подходом.

### Шаг 7. Гибридный поиск (RRF)

Что делается:

- объединяются ранги BM25 и dense через `RRF` (Reciprocal Rank Fusion).

Зачем:

- BM25 и dense закрывают разные типы запросов;
- RRF объединяет их без сложной калибровки шкал score.

### Шаг 8. Дополнительное переранжирование (опционально)

Что делается:

- добавляется бонус по типу датасета (`typical`/`product`);
- учитывается эко-приоритет: меньший `GWP` — выше при прочих равных.

Зачем:

- в реальном применении важна не только текстовая релевантность, но и прикладные критерии выбора материала.

### Шаг 9. Оценка качества

Что считается:

- `Recall@1`, `Recall@5`, `Recall@10`, `Recall@20`.

Что сравнивается:

1. базовые retrieval-стратегии (BM25, dense, гибрид RRF);
2. гибрид с дополнительной логикой (type bonus, type+GWP).

Почему это важно:

- отдельно видно, где заканчивается «чистый retrieval» и начинается «бизнес-логика ранжирования».

### Шаг 10. Чат-интерфейс (демо)

Что реализовано:

- интерактивный интерфейс на Gradio;
- в ответе выводятся `top-5` материалов;
- по каждому материалу показываются объяснимые поля (score, `dataset_type`, `GWP`, `UUID`, категория).

Зачем:

- это удобный формат для ручной проверки качества и демонстрации системы.

## LLM-слой: что делает и чего не делает

В проекте есть LLM-слой для интерпретации результата поиска.

Что он делает:

- берет только `top-5` материалов, которые вернул retrieval-пайплайн;
- получает структурированный контекст (`name`, `category`, `dataset_type`, `unit`, `gwp`, `uuid`);
- формирует краткое объяснение (почему материалы релевантны, какие компромиссы по `GWP`).

Что он принципиально не делает:

- не ищет новые объекты вне retrieval-выдачи;
- не подменяет ранжирование и отбор кандидатов;
- не должен придумывать факты: в промпте явно задано `Use only provided context`.

Идея архитектуры:

- retrieval отвечает за **фактическую выборку**;
- LLM отвечает за **читаемую интерпретацию** этой выборки.

Это важно для надежности: если LLM недоступен или ошибается, сама retrieval-часть продолжает работать и выдавать корректные `top-k` записи.

## Как именно происходит отбор, threshold и сортировка

Ниже описан фактический порядок из `recommend_materials(...)`.

### 1) Базовый кандидатный пул (`top_candidates`)

Сначала строится список кандидатов размером `top_candidates` одним из методов:

- `bm25` -> `bm25_score`;
- `vector` -> `similarity` (cosine/IP по нормализованным эмбеддингам);
- `hybrid` -> `rrf_score` (слияние BM25 + dense через RRF).

Это «черновой» пул для дальнейшей фильтрации.

### 2) Semantic gate с динамическим порогом (threshold)

Если `use_semantic_gate=True`, запускается отдельный пробный dense-поиск (`probe_k`) и вычисляется порог:

- `tau = max(tau_abs, top1_similarity - delta)`.

Смысл параметров:

- `tau_abs` — нижняя абсолютная граница качества (например, `0.50`);
- `delta` — насколько можно отступить от лучшего кандидата `top1_similarity`;
- `probe_k` — сколько объектов анализировать для оценки «плотности» релевантных соседей.

Дальше в gate проходят только кандидаты с `similarity >= tau`.

### 3) Стабилизация размера gate-пула

Чтобы gate не был слишком узким или слишком широким, применяются границы:

- минимум `gate_n_min` (если прошло мало, добираем из начала probe-списка);
- максимум `gate_n_max` (если прошло слишком много, обрезаем хвост).

Так получаем адаптивный, но контролируемый набор `allowed_uuids`.

### 4) Fallback-защита

После фильтрации проверяется размер списка:

- если после gate осталось меньше, чем `top_k`, используется исходный `base_before_gate`.

Зачем это нужно:

- чтобы не потерять качество UX и не вернуть слишком короткий ответ;
- чтобы агрессивный threshold не «сломал» выдачу на сложных запросах.

### 5) Сортировка по релевантности + type bonus

Для каждого кандидата считается:

- `score_with_type = base_score + type_bonus`.

Где `base_score` — это одна из величин (`bm25_score` / `similarity` / `rrf_score`),  
а `type_bonus` зависит от `mode` и `dataset_type`.

Первичная сортировка:

- по `score_with_type` по убыванию.

### 6) Финальный rerank по GWP (опционально)

Если `use_gwp_rerank=True`, применяется вторичная сортировка внутри уже отобранного пула:

1. сначала записи с валидным `gwp` (NaN уходят вниз),
2. затем по возрастанию `gwp` (меньше — лучше),
3. при равенстве — по убыванию `score_with_type`.

Это означает, что `GWP` работает как приоритет ранжирования, но не отменяет полностью базовую релевантность.

### 7) Что попадает в ответ пользователю

В ответ возвращается:

- `ranked[:top_k]`.

То есть итоговый список — это результат всех этапов:

- retrieval -> semantic gate (с threshold) -> type bonus -> GWP rerank -> `top_k`.

### 8) Мини-пример на числах (как меняется порядок)

Допустим:

- `top_k=3`
- `tau_abs=0.50`
- `delta=0.08`
- у probe-листа лучший `top1_similarity=0.62`

Тогда threshold:

- `tau = max(0.50, 0.62 - 0.08) = 0.54`

Исходные кандидаты после базового retrieval (пример):

| Кандидат | similarity | dataset_type      | type_bonus | gwp  |
|----------|------------|-------------------|-----------:|-----:|
| A        | 0.62       | specific dataset  | 0.00       | 18.0 |
| B        | 0.60       | generic dataset   | 0.00       |  9.0 |
| C        | 0.58       | product dataset   | 0.05       | 14.0 |
| D        | 0.55       | product dataset   | 0.05       | 11.0 |
| E        | 0.53       | product dataset   | 0.05       |  6.0 |
| F        | 0.49       | product dataset   | 0.05       |  4.0 |

Шаг 1. Semantic gate (`similarity >= 0.54`):

- проходят: `A, B, C, D`
- отсекаются: `E, F` (даже если у них низкий `gwp`, они недостаточно релевантны по смыслу)

Шаг 2. Добавление `type_bonus` и сортировка по `score_with_type`:

- `A: 0.62 + 0.00 = 0.62`
- `B: 0.60 + 0.00 = 0.60`
- `C: 0.58 + 0.05 = 0.63`
- `D: 0.55 + 0.05 = 0.60`

Порядок после этого шага: `C (0.63)`, `A (0.62)`, `B (0.60)`, `D (0.60)`.

Шаг 3. Опциональный `GWP rerank` внутри прошедшего пула:

- сортируем по `gwp` (меньше лучше), tie-break по `score_with_type`;
- получаем: `B (9.0)`, `D (11.0)`, `C (14.0)`, `A (18.0)`.

Итоговый ответ при `top_k=3`:

- `B`, `D`, `C`.

Что показывает пример:

- `threshold` защищает от попадания слаборелевантных объектов;
- `type_bonus` помогает продвинуть приоритетный тип датасета;
- `GWP` переупорядочивает только внутри уже релевантного semantic-пула.

## Как запустить

### 1) Установка зависимостей

Рекомендуется Python 3.10+.

```bash
pip install -r requirements.txt
```

Если запускаете в Colab, можно установить зависимости в ячейке ноутбука.

### 2) Подготовка данных

Откройте и выполните ноутбук:

- `RAG_data_preparation_from_obd.ipynb`

После выполнения должны появиться/обновиться:

- `data/df_text.csv`
- `data/df_numeric.csv`
- `data/eval_queries.csv`
- `data/eval_qrels.csv`

### 3) Retrieval, оценка и демо

Откройте и выполните ноутбук:

- `RAG_material_search_oekobaudat.ipynb`

Рекомендуемый порядок ячеек:

1. импорт и загрузка подготовленных данных;
2. построение эмбеддингов и индексов FAISS;
3. функции BM25 / dense / hybrid retrieval;
4. benchmark и графики;
5. тестовые запросы;
6. запуск Gradio-чата.

## Ограничения и идеи для улучшения

- `eval_qrels` сформирован по правилам; разметка экспертами повысит надежность оценки;
- переранжирование только по `GWP` упрощает задачу; полезно добавить многокритериальную модель;

- LLM-слой для интерпретации уже реализован; следующий шаг — повысить надежность (fallback при недоступности API, кэш ответов, контроль шаблона объяснения, пост-проверка фактов по полям контекста).
